## 1. 文档信息

| 项目 | 内容 |

| 文档版本 | v1.0.0 |
| 创建日期 | 2026-02-13 |
| 更新日期 | 2026-03-26 |
| 文档状态 | 与当前本地实现对齐 |

## 2. 设计范围

本文档描述当前本地实现中的前端状态管理、认证方式、路由保护、项目/任务/批量/下载链路，以及后端 ownership、任务模型和即梦 Session 解析设计。

当前设计不再沿用旧文档中的以下说法：

- localStorage 键为 `session_id`
- 注册只传邮箱和密码
- 所有普通用户可见相同项目列表
- 批量任务是单层任务直接执行
- 浏览器直接读取服务器本地视频路径

## 3. 前端详细设计

### 3.1 App.tsx

#### 3.1.1 组件职责

`src/App.tsx` 负责：

- 初始化当前登录用户
- 通过 `ProtectedRoute` 执行登录保护与管理员保护
- 通过 `MainLayout` 装配侧边栏与主区域
- 将 `currentUser` 传入 `AppProvider`

#### 3.1.2 路由设计

当前有效路由：

- `/login`
- `/register`
- `/`
- `/batch`
- `/download`
- `/settings`
- `/admin`
- `* -> /`

保护规则：

- 未登录访问受保护路由时跳转 `/login`
- 非管理员访问 `/admin` 时跳回 `/`
- 已登录用户访问 `/login` 或 `/register` 时跳转回首页

### 3.2 认证状态与本地存储

#### 3.2.1 平台登录会话

`src/services/authService.ts` 当前行为：

- 登录会话存储在 `seedance_session_id`
- 用户缓存存储在 `seedance_user_cache`
- 鉴权头为 `X-Session-ID`
- 注册与登录成功后都会写入 session 和 user cache
- `logout()` 和会话失效都会清理本地缓存

#### 3.2.2 认证服务接口

前端主要接口：

- `register(credentials)`
- `login(credentials)`
- `logout()`
- `getCurrentUser()`
- `changePassword(oldPassword, newPassword)`
- `checkIn()`
- `sendEmailCode(email)`
- `verifyEmailCode(email, code)`

当前注册凭证结构来自 `src/types/index.ts`：

```ts
interface RegisterCredentials {
  email: string;
  password: string;
  emailCode: string;
}
```

因此旧文档里“注册不需要 `emailCode`”的描述已经失效。

### 3.3 AppContext.tsx

#### 3.3.1 状态结构

`src/context/AppContext.tsx` 使用 `useReducer` 管理：

- `projects`
- `currentProject`
- `tasks`
- `currentTask`
- `settings`
- `loading`
- `error`

#### 3.3.2 登录切换时的状态重置

`AppProvider` 直接接收 `currentUser`，并在用户为空时执行状态清空：

- 清空 `projects`
- 清空 `currentProject`
- 清空 `tasks`
- 清空 `currentTask`
- 清空 `error`
- 结束 `loading`
- 把 `settings` 重置为 `{}`

此外，`currentUser` 变化后还会重新触发：

- `loadProjects()`
- `loadSettings()`

这保证了切换账号后不会继续看到上一个账号的项目、任务或设置。

### 3.4 项目与任务服务层

#### 3.4.1 projectService.ts

`src/services/projectService.ts` 当前所有项目接口都显式带 `getAuthHeaders()`：

- `getProjects()`
- `getProject(id)`
- `createProject(...)`
- `updateProject(id, data)`
- `deleteProject(id)`
- `getProjectTasks(id, options)`

`getProjectTasks` 支持前端传入：

- `status`
- `taskKind`

#### 3.4.2 taskService.ts

`src/services/taskService.ts` 管理任务创建、更新、删除、素材和生成。当前前端任务模型已支持：

- `taskKind`
- `rowIndex`
- `videoCount`

单任务生成接口可返回：

- `batchId`
- `totalTasks`
- `outputTaskIds`

这意味着当前单任务生成也可能进入由后端统一调度的批量/展开语义，而不是旧版单条任务直接出结果的模型。

### 3.5 batchService.ts

#### 3.5.1 批量启动

`startBatchGenerate(projectId, taskIds, options)` 通过 `/api/batch/generate` 发起批量执行。

异常设计：

- 当后端返回无效任务时，前端抛出 `BatchServiceError`
- `BatchServiceError.invalidTasks` 用于界面展示不合法草稿任务原因

#### 3.5.2 批量状态与操作

以下接口都显式要求登录态：

- `getBatchStatus(batchId)`
- `pauseBatch(batchId)`
- `resumeBatch(batchId)`
- `cancelBatch(batchId)`
- `collectBatchVideos(batchId)`

### 3.6 downloadService.ts

#### 3.6.1 下载列表与服务端下载

`src/services/downloadService.ts` 当前提供：

- 下载任务分页列表
- 服务端下载单任务视频
- 批量下载视频到服务器本地目录
- 刷新下载列表
- 从即梦同步结果
- 打开本地文件夹

这些接口都显式使用 `getAuthHeaders()`。

#### 3.6.2 浏览器下载本地文件

当前浏览器下载不是直接读本地路径，而是两步：

1. `POST /api/download/tasks/:id/file-token`
2. `GET /api/download/file-by-token?token=...`

前端实现会先获取 token，再构造浏览器下载链接并触发 `<a download>` 行为。

### 3.7 settingsService.ts

#### 3.7.1 全局设置接口

当前前端 `getSettings()` / `updateSettings()` 请求 `/api/settings` 时，没有显式调用 `getAuthHeaders()`。

这部分在写文档时必须按“当前前端调用链”表述，不能笼统写成“所有 settings 接口统一显式携带认证头”。

#### 3.7.2 Session 账号接口

以下接口显式要求登录态：

- `getSessionAccounts()`
- `createSessionAccount(input)`
- `updateSessionAccount(id, input)`
- `deleteSessionAccount(id)`
- `setDefaultSessionAccount(id)`
- `testJimengSessionId(sessionId)`

返回结构包含：

- `accounts`
- `effective`

其中 `effective` 表示当前实际生效的即梦 Session 解析结果。

### 3.8 videoService.ts

#### 3.8.1 当前调用链

`src/services/videoService.ts` 中：

- 提交 `/api/generate-video` 时显式带 `getAuthHeaders()`
- 轮询 `/api/task/:taskId` 时当前前端未显式附加认证头

因此视频链路的鉴权说明也必须按现状分别描述，不能简单写成“视频相关接口全部统一鉴权”。

## 4. 后端详细设计

### 4.1 认证与管理员能力

#### 4.1.1 认证职责

后端认证体系负责：

- 注册、登录、登出
- 会话校验
- 改密
- 邮箱验证码发送与验证
- 积分扣减 / 充值 / 签到
- 管理员统计与用户管理

#### 4.1.2 会话校验边界

平台侧鉴权核心基于 `X-Session-ID`。当前业务文档应统一描述为：

- 会话通过请求头传递
- 会话不出现在 URL 中
- 普通用户能力由当前会话绑定的 user 决定
- 管理员接口需额外满足管理员角色校验

### 4.2 ownership 与项目隔离

#### 4.2.1 项目查询

`server/services/projectService.js` 的 `getAllProjects(userId, isAdmin)` 当前逻辑：

- 管理员可取全量项目
- 普通用户查询时附加 `WHERE p.user_id = ?`

#### 4.2.2 项目下任务访问

`getTasksByProjectId(projectId, options, userId, isAdmin)` 在非管理员场景会先检查项目归属：

- 若项目不存在或不属于当前用户，直接报错
- 普通用户不能通过项目入口访问其他用户任务

这部分是当前“谁创建，谁可见”的基础收口点之一。

### 4.3 任务模型与 draft/output 展开

#### 4.3.1 tasks 表扩展字段

当前任务模型关键字段：

- `task_kind`
- `source_task_id`
- `row_group_id`
- `row_index`
- `video_count`
- `output_index`

#### 4.3.2 draft 展开逻辑

`server/services/taskService.js` 中 `expandDraftTaskWithDb` 的核心行为：

- 只允许 `task_kind === 'draft'` 的任务被展开
- 根据 `video_count` 生成多个 output 任务
- output 任务继承 draft 的 `project_id`、`user_id`、`prompt`、素材和账号信息
- 通过 `source_task_id` 与 `row_group_id` 保持关联

这构成了当前批量执行的基础数据结构。

### 4.4 批量调度器

#### 4.4.1 任务快照

`server/services/batchScheduler.js` 在运行时维护任务快照，包括：

- `taskId`
- `prompt`
- `status`
- `progress`
- `errorMessage`
- `submitId`
- `historyId`
- `itemId`
- `videoUrl`
- `sourceTaskId`
- `rowGroupId`
- `outputIndex`
- `assetCount`

#### 4.4.2 ownership 透传

`buildBatchDetail(batch, runtime, userId, isAdmin)` 与 `start(batchId, taskIds, userId, isAdmin)` 都透传当前用户上下文。

因此：

- 普通用户查询批量详情时只能看到自己有权访问的任务
- 管理员可以查看全量批量信息
- 批量状态、暂停、恢复、取消等衍生能力也继承 ownership 规则

#### 4.4.3 Session 缺失提示

批量执行前如果没有可用 Session，会显式抛出：

- “未配置可用的 SessionID，请在设置页添加并设为默认账号”

这说明批量链路已经依赖用户自己的即梦 Session 解析结果，而不是旧版的单全局 Session 假设。

### 4.5 即梦 Session 账号管理

#### 4.5.1 数据规则

`server/services/jimengSessionService.js` 实现：

- 同一用户下 `session_id` 唯一
- 第一个账号自动设为默认
- 设置默认时会清空该用户其他账号的默认标记
- 删除默认账号后会自动补一个新的默认账号

#### 4.5.2 生效 Session 解析

`resolveEffectiveSession(userId)` 返回：

- `source`
- `sessionId`
- `account`

当前优先级固定为：

1. `user_default`
2. `legacy_global`
3. `env_default`
4. `none`

其中 `settings.session_id` 仅作为 `legacy_global` 兼容项存在，不应再作为主流程产品叙事。

### 4.6 下载链路

#### 4.6.1 服务端下载到本地

下载管理相关后端职责包括：

- 根据任务或批量下载视频到本地目录
- 维护下载状态与下载路径
- 提供历史同步能力
- 提供打开文件夹能力

#### 4.6.2 token 化浏览器下载

当前浏览器下载链路的设计目标是：

- 浏览器不直接感知服务器本地真实路径
- 通过一次性 token 间接获取文件
- 将 token 与当前用户上下文绑定，保证归属隔离

## 5. 当前接口边界

### 5.1 明确走前端显式认证头的调用

当前前端调用层面，以下接口明确通过 `getAuthHeaders()` 发送 `X-Session-ID`：

- `projectService.ts` 全部项目接口
- `taskService.ts` 全部任务 / 素材 / 生成相关接口
- `batchService.ts` 全部批量接口
- `downloadService.ts` 全部下载管理接口
- `settingsService.ts` 的 `session-accounts` 相关接口
- `videoService.ts` 提交 `/api/generate-video`

### 5.2 需要谨慎描述的边界

当前前端代码中以下边界不能被文档误写：

- `settingsService.ts` 的 `getSettings()` / `updateSettings()` 未显式加 `getAuthHeaders()`
- `videoService.ts` 轮询 `/api/task/:taskId` 未显式加 `X-Session-ID`

## 6. 数据库关键设计

### 6.1 ownership 相关字段

当前最关键的归属字段包括：

- `projects.user_id`
- `tasks.user_id`
- `jimeng_session_accounts.user_id`

它们共同支撑普通用户只见自己数据、管理员可见全量数据的权限模型。

### 6.2 兼容项与增强字段

相对旧文档，以下表或字段是当前实现必须覆盖的：

- `email_verification_codes.purpose`
- `email_verification_codes.code_hash`
- `email_verification_codes.salt`
- `email_verification_codes.attempts`
- `email_verification_codes.request_ip`
- `email_verification_codes.consumed_at`
- `system_config`
- `jimeng_session_accounts`
- `settings.session_id`（legacy 兼容项）

## 7. 非功能设计约束

### 7.1 安全

- 平台登录 Session 不出现在 URL 中
- 普通用户不能访问他人项目、任务、批量或下载记录
- 浏览器下载本地文件必须通过 token 化链路
- 管理员能力必须显式受权限控制

### 7.2 可维护性

后续修改认证、路由、ownership、Session 账号或下载设计时，优先核对：

1. `src/App.tsx`
2. `src/context/AppContext.tsx`
3. `src/services/authService.ts`
4. `src/services/settingsService.ts`
5. `src/services/downloadService.ts`
6. `server/services/projectService.js`
7. `server/services/taskService.js`
8. `server/services/jimengSessionService.js`
9. `server/services/batchScheduler.js`
10. `server/database/schema.sql`

## 8. 结论

当前 Seedance 2.0 的详细实现已经明确是一个带有：

- 用户认证与管理员后台
- ownership 隔离
- 多页面工作台
- AppContext 状态重置
- draft/output 双层任务模型
- 多即梦 Session 账号解析
- token 化下载链路

的完整系统，而不是早期单页面、单 SessionID、弱权限隔离的生成工具。
