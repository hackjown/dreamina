## 2. 设计目标

本文档以当前本地代码实现为准，描述 Seedance 2.0 的整体架构、模块划分与关键链路。它不再沿用旧 GitHub 时代“单页面 + 单 SessionID + 单层任务模型”的系统叙述。

当前系统目标：

- 为登录用户提供单任务生成、项目管理、批量生成、下载管理的一体化工作台
- 对普通用户执行严格的 ownership 隔离，对管理员保留全量可见能力
- 通过多即梦 Session 账号管理提升生成稳定性与可维护性
- 通过 draft/output 双层任务模型支持批量调度和多输出展开
- 通过服务端保存 + token 化浏览器下载保证下载链路安全

## 3. 系统概览

### 3.1 产品形态

Seedance 2.0 是一个 React + TypeScript + Vite 前端与 Express + SQLite 后端组成的 AI 视频生成系统。用户在网页端完成认证、项目与任务管理、批量调度和下载管理，后端负责权限校验、任务落库、即梦能力调用和视频保存。

### 3.2 当前版本的关键变化

相对旧版说明，当前架构已经明确演进为：

- 类型入口为 `src/types/index.ts`，不再是 `src/types.ts`
- 平台登录会话与即梦 Session 账号分离，形成双会话模型
- 路由为多页面工作台，而非单页模式
- 项目 / 任务 / 批量 / 下载记录按用户归属隔离
- 任务模型为 `draft -> output*` 双层结构
- 下载链路为服务端落盘后再通过一次性 token 提供给浏览器
- `AppProvider` 会随着 `currentUser` 变化清空并重载全局状态

## 5. 双会话模型

### 5.1 平台登录会话

平台登录会话由应用自身用户体系管理：

- localStorage 键：`seedance_session_id`
- 用户缓存键：`seedance_user_cache`
- 请求头：`X-Session-ID`
- 主要入口：`src/services/authService.ts`

这套会话用于平台鉴权、页面路由保护、项目/任务/批量/下载权限控制。

### 5.2 即梦 Session 账号

即梦 Session 账号由登录用户在设置页维护，用于后端访问即梦平台：

- 数据表：`jimeng_session_accounts`
- 前端入口：`src/services/settingsService.ts`
- 后端解析：`server/services/jimengSessionService.js`

当前有效 Session 解析优先级：

1. `user_default`
2. `legacy_global`
3. `env_default`
4. `none`

## 6. 前端概要设计

### 6.1 路由结构

当前前端路由来自 `src/App.tsx`：

- `/login`
- `/register`
- `/`
- `/batch`
- `/download`
- `/settings`
- `/admin`
- `* -> /`

路由规则：

- 未登录用户访问受保护路由时跳转到 `/login`
- `requireAdmin` 路由仅允许管理员访问
- 业务页面统一包裹在 `MainLayout` 中，由 `Sidebar + main` 组成

### 6.3 前端状态模型

全局状态由 `src/context/AppContext.tsx` 管理，核心状态包括：

- `projects`
- `currentProject`
- `tasks`
- `currentTask`
- `settings`
- `loading`
- `error`

关键行为：

- `AppProvider` 接收 `currentUser`
- 当 `currentUser` 为空时，清空项目、任务、错误和 loading，并把 `settings` 重置为空对象
- 当 `currentUser` 变化时，重新加载项目和设置

该设计用于避免登录切换后保留上一个账号的项目或设置上下文。

## 8. 关键架构约束

### 8.1 ownership 隔离

系统强制执行“谁创建，谁可见”的数据归属规则：

- 普通用户只能查看自己的项目、任务、批量和下载记录
- 管理员可以查看全量数据
- 批量详情、批量状态、批量暂停/恢复/取消、下载列表都必须沿用同一归属约束

### 8.2 任务模型

当前任务模型不是旧版单层任务，而是：

- `draft`：编辑态任务，承载 prompt、素材、行号、计划输出数
- `output`：从 draft 展开的实际生成任务

关键字段：

- `task_kind`
- `source_task_id`
- `row_group_id`
- `row_index`
- `video_count`
- `output_index`

### 8.3 下载模型

下载管理分为两段：

1. 服务端把视频保存到本地目录
2. 浏览器通过一次性 token 访问 `file-by-token` 完成下载

这意味着当前架构不是“前端直接读取服务器本地路径”。

## 9. 核心业务流程

### 9.1 登录与初始化

```text
用户登录/注册
  -> authService 保存 seedance_session_id 和 seedance_user_cache
  -> App.tsx 设置 currentUser
  -> AppProvider 重新加载项目和设置
  -> ProtectedRoute 放行受保护页面
```

### 9.2 单任务与项目任务管理

```text
用户选择项目
  -> 加载项目任务
  -> 创建或编辑 draft 任务
  -> 上传图片/音频素材
  -> 发起生成或加入批量链路
```

### 9.3 批量生成

```text
选择项目下 draft 任务
  -> 后端校验 ownership / draft 类型 / prompt / 图片素材
  -> 展开 output 任务
  -> BatchScheduler 按并发与间隔执行
  -> 返回运行时快照与 invalidTasks
```

### 9.4 下载管理

```text
用户进入下载管理
  -> 查询下载列表
  -> 服务端下载单任务或批量视频到本地
  -> 前端申请 file-token
  -> 浏览器访问 file-by-token 下载本地文件
```
